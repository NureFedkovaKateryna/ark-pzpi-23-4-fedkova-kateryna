2 ЗАВДАННЯ
Метою даної роботи є навчитися розгортати програмну систему, перевіряти її функціональність, виконувати налаштування, а також демонструвати та документувати роботу системи.
3 ОПИС ВИКОНАНОЇ РОБОТИ
Розгортання системи здійснюється з використанням технології контейнеризації Docker, що дозволяє забезпечити ізольоване, відтворюване та незалежне від середовища виконання розгортання застосунку. Для локальної розробки та запуску сервісів використовується Docker Compose. Система побудована на основі веб фреймворку Django та запускається у Python-середовищі версії 3.11.4.
Для створення Docker-образу використовується файл Dockerfile. Базовим образом обрано офіційний образ python:3.11.4-slim-buster, який забезпечує мінімальний розмір контейнера та необхідне середовище для запуску Python-застосунку. Основні етапи налаштування Docker-образу:
* встановлення робочої директорії контейнера /usr/src/app;
* налаштування змінних середовища (для заборони створення .pyc файлів та забезпечення негайного виводу логів у консоль);
* оновлення менеджера пакетів pip;
* встановлення залежностей із файлу requirements.txt;
* копіювання вихідного коду проєкту в контейнер;
* налаштування запуску застосунку за допомогою WSGI-сервера Gunicorn, який прослуховує порт 10000.
Таким чином, Docker-образ містить усі необхідні залежності для коректної роботи Django-застосунку.
Для керування контейнерами використовується файл docker-compose.yaml, у якому описано сервіс web. Основні параметри сервісу:
* build - створення образу на основі Dockerfile;
* command - запуск Django-вбудованого сервера розробки за адресою 0.0.0.0:8000;
* volumes - монтування локальної директорії проєкту в контейнер для автоматичного оновлення коду без перескладання образу;
* ports - переброс порту 8000 контейнера на порт 8000 хост-машини;
* env_file - підключення змінних середовища з файлу .env.dev.
Використання Docker Compose дозволяє централізовано керувати конфігурацією сервісів та значно спрощує процес локального запуску і тестування системи.
Для розгортання серверної частини системи було обрано хмарну платформу Render. Основними причинами такого вибору є простота налаштування, підтримка Docker-контейнерів та автоматизація процесу деплою. Платформа Render дозволяє:
* розгортати веб застосунки без необхідності ручного налаштування серверів;
* використовувати Docker-образи, що забезпечує однакову поведінку системи в локальному та продакшн-середовищах;
* автоматично перезапускати сервіс у разі збоїв;
* керувати змінними середовища через веб інтерфейс.
Серверна частина системи реалізована на основі фреймворку Django, який запускається у контейнері з використанням WSGI-сервера Gunicorn.
Для зберігання даних система використовує реляційну базу даних PostgreSQL, яка добре інтегрується з Django через ORM. Причини вибору реляційної бази даних:
* підтримка складних зв’язків між сутностями;
* забезпечення цілісності даних;
* зручна міграція структури бази даних;
* надійність та масштабованість.
Платформа Render надає можливість використання керованої бази даних, що зменшує складність адміністрування.
Для взаємодії між клієнтом і сервером використовується протокол HTTP/HTTPS, який є стандартом для веб застосунків. Основні переваги використання HTTP/HTTPS:
* сумісність з веб браузерами та мобільними клієнтами;
* підтримка REST API;
* можливість захищеного передавання даних через HTTPS;
* простота інтеграції з зовнішніми сервісами.
Обмін даними здійснюється у форматі JSON, що є легким, читабельним та зручним для обробки.
Використання Docker дозволяє ізолювати застосунок разом із залежностями та середовищем виконання. Це значно зменшує ризик помилок, пов’язаних із відмінностями середовищ.
Обраний стек технологій (Docker, Django, Gunicorn, Render, HTTP/HTTPS) забезпечує надійне, масштабоване та зручне розгортання системи.
